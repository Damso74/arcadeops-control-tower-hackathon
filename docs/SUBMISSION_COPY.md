## Long description

Autonomous AI agents are now deployed in production — but they remain
black boxes. Teams cannot easily answer the only question that matters
before scaling them up: **is this run actually safe to ship?**

**ArcadeOps Control Tower** turns every autonomous agent run into an
auditable trace and lets a model — **Google Gemini** — act as a
reliability judge over that trace.

The demo includes an unsafe CRM-agent scenario. Gemini blocks the run,
explains the risk, recommends production guardrails, then re-scores the
run as a what-if remediation simulation. The judge is honest: re-scoring
does not always return "ready" — residual risks stay visible.

The hackathon demo is structured as three layers:

1. **Trace ingestion layer.** Three input modes, one judge:
   - **Recorded scenarios** — Blocked CRM write agent (critical),
     Needs-review support agent (medium), Production-ready research
     agent (low). Each ships with a textual trace, an evidence card
     list, and snapshot observability metrics.
   - **Deterministic SSE replay** of a sanitized real agent run,
     for video reproducibility (no API key required).
   - **Pasted trace** — free-form text up to 12 000 characters,
     sanitized server-side (emails, URLs, bearer tokens, API keys,
     UUIDs all redacted before the prompt is built).
2. **Gemini Reliability Judge.** When `GEMINI_API_KEY` is configured,
   Gemini 2.5 Flash reads the trace and returns a strict JSON verdict:
   - `readinessScore` (0–100),
   - `verdict` (`ready` / `needs_review` / `blocked`),
   - typed `risks` (severity × category × evidence),
   - cost / tool-safety / observability assessments,
   - `missingEvidence` (what a real reviewer would still need),
   - a numbered `remediationPlan`,
   - an `executiveDecision` and a `businessValue` paragraph.

   On top of this, **Simulate guardrails and re-score** runs the judge
   a second time on the same trace, this time as a what-if remediation
   simulation. The before/after readiness comparison is the wow moment
   of the demo: a critical run can move from BLOCKED to READY WITH
   MONITORING — or stay BLOCKED if Gemini still finds residual risk.

   The demo combines Gemini's reasoning with **deterministic production
   policy gates**: destructive actions without approval, outbound
   customer messages without review, and write actions without
   replay/audit evidence cannot be marked production-ready, even if a
   model summarizes the run. Gemini provides the audit; ArcadeOps
   enforces the non-negotiable rules.
3. **Deployment layer.** Public demo on Vercel. Dockerfile + `/api/health`
   + standalone Next.js bundle make the same image trivially deployable
   to **Vultr** (VPS, Container Registry, or Vultr Kubernetes Engine).
   Replay works with zero env vars; the Gemini judge is enabled by
   simply adding the key. The judge endpoint is rate-limited (5 req /
   10 min / IP) and pasted traces are hard-capped to 12 000 characters
   and sanitized server-side.

Behind the scenes, the same Control Tower event model is also wired to
the live ArcadeOps backend through a server-side proxy. That live mode
is **intentionally disabled in the public deployment** for safety: the
production ArcadeOps platform handles real customer data. The replay
trace published in this repo is a **sanitized** projection — no orgIds,
no userIds, no internal URLs, no client names — generated by an export
script in the private repo.

The goal is simple: help teams trust, debug and scale autonomous agents
safely, by giving every run a flight recorder and a second opinion.

## Sponsor prize fit

### Google — Best use of Gemini (PRIMARY TARGET)

Gemini is **not a chatbot wrapper** in this project. It is the **decision
engine** — a structured-JSON reliability judge over an agent trace. The
model:

- consumes a typed `JudgeRunSnapshot` (mission, phases, tool calls,
  observability metrics, risk flags, final result),
- runs in a Node.js serverless function with `responseMimeType: "application/json"`
  and `temperature: 0.2` to keep verdicts reproducible,
- returns a strict schema the UI renders directly: readiness score,
  verdict, risks (severity × category × evidence), cost / tool-safety /
  observability assessments, missing evidence, remediation plan,
  executive decision, business value.

The judge replaces what is today a **manual reviewer task** for any
team running autonomous agents in production. It is the differentiator
versus generic agent demos.

### Vultr — Best use of Vultr (SECONDARY)

The repo ships:

- a multi-stage **Dockerfile** producing a Next.js 16 standalone image,
- a non-root container, `.dockerignore` excluding all secrets,
- a **`/api/health`** endpoint with no external dependencies,
- a step-by-step **`docs/VULTR_DEPLOYMENT.md`** covering single-VPS
  Docker, plain Node 20, and Vultr Kubernetes Engine paths.

ArcadeOps Control Tower is **production infrastructure for AI agents** —
the natural fit for predictable, controlled VPS hosting. We will switch
the public demo to a Vultr VPS once the Vercel build remains stable for
several days.

### Others — explicitly out of scope

Kraken trading, Speechmatics and other infra-heavy sponsor tracks are
**not pursued** to keep the project focused on the Gemini reliability
story.

## What is real

- The Next.js application, the `/control-tower` UI, and the SSE event
  model are real.
- The `/api/replay` deterministic stream is real and identical between
  runs (replays produce the same trace — by design).
- The **Gemini Reliability Judge** is fully real when `GEMINI_API_KEY`
  is configured. The judge runs against the replayed trace and returns
  a live verdict on every click.
- The live ArcadeOps backend adapter has been **tested end-to-end
  locally** against the private platform but is disabled in the public
  deployment for safety. See `src/app/api/arcadeops/run/route.ts`.
- The replay fixture is generated by a sanitized export script in the
  private ArcadeOps repo (`scripts/export-control-tower-trace.ts`).

## What is replayed (deterministic, by design)

- LLM execution, web searches, browse calls, evaluation and report
  generation — all replayed from the recorded trace so judging stays
  reproducible.

The replay disclosure is explicit in the README, in the page footer,
and in the `Run again` micro-copy under the Result panel.
